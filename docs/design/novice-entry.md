# 設計：素人主入口 + 把關綁預覽（可執行草案）

> 狀態：**設計草案（尚未實作）**。來源：`ux-product-notes.md` §5 裁決的落地。日期 2026-06-27。
> 一句話：把素人的「第一次成功」做順——首屏只給一個溫暖入口、術語延後揭露；而且讓素人的「把關」永遠面對**看得懂的東西**（預覽＋白話），不是程式碼。
> 定位：**幾乎全是資訊架構（IA）+ 文案 + 條件顯示**，不重寫核心 pipeline、不動治理 runtime。低風險、可分批出。

---

## 0. 要解的問題（來自 UX 討論）

最大 UX 風險＝**系統一開始就逼素人進入開發者心智模型**（git/PR/token/治理/證據鏈）。
最大設計盲點＝**素人看不懂程式碼草稿 → 盲目按下一步 → human-in-the-loop 淪為形式**。

兩件事對應兩個設計原則：
1. **漸進揭露**：素人首屏只需要「描述想法 / 幫我想想」；開發者向的東西（連 git、匯入、成品類型、開發強度）收進「進階 / 我已有專案」。
2. **把關綁預覽**：素人的審查與核准，一律面向**線上預覽 + EXPLAINER 白話導覽 + 白話計畫**；**永不要素人面對原始碼 / diff / 治理事件**（那是開發者門的東西）。

> 明確 **不做**（沿用 §5 裁決）：不做首屏 NLP 自動判斷意圖分流（會誤判、且與「不靠 AI 亂猜」氣質衝突）；不自編金流／精確金額（誠實鐵律）。

## 1. 首頁（`WebUiController` index，現況 line ~83–143）

### 現況
- 已有：粉圓吉祥物、白話開場、Discovery CTA「🤔 還不知道要做什麼？」(line 87–91)。
- 但同一張表單**並列**：模式 new/import/**連 git**(line 96–99)、成品類型 5 選 1(line 122–128)、開發強度(line 132–136)。對素人＝一次看到太多考題（尤其「🔧 連 git 專案」）。

### 改法（IA + 條件顯示）
- **主區（素人預設可見）**：開場文案 + Discovery CTA + 標題 + 詳細描述 + 「🚀 開始」。`mode` 預設 `new`、`projectType` 預設 `recommend`（智慧推薦）、`strength` 預設 `3`（穩健）——**全部沿用現有預設值，但從首屏收起**。
- **次要區（安靜的進階揭露）**：一個低調可展開區塊，標籤「**⚙️ 我已有專案，或想自己設定**」。展開後才出現：模式 import/existing + 上傳 zip(line 104–108) + 成品類型 + 開發強度。
  - 命名用**任務語言**，不要寫「素人／開發者」。
  - ⚠️ **收合狀態的標籤（`<summary>`）本身一定看得見**，所以**標籤裡不可列出 git／zip／成品類型／強度等術語**——否則「藏起來」形同虛設（這正是首版踩到、Codex stop-time review 抓到的雷）。標籤只給中性任務語言，術語留在展開後的內容裡。
  - 進階區永遠可展開／收合；切換不重置已填內容。
- **共同品牌脊椎**：主區與進階區都保留「**最後由你把關、由你決定**」這句（line 86 已有，勿移除）——兩條路徑都看得到「人類是最後一關」。

### 驗收
- 素人首屏**不出現** git / zip / 開發強度 / 成品類型字樣（收在進階）。
- 素人只靠「打一句描述 + 開始」或「幫我想想」即可前進，不需做任何技術選擇。
- 開發者點一下「進階／我已有專案」就拿回完整控制，內容不流失。

## 2. 開工前確認頁（`TaskController` confirm 視圖，現況 line ~624–676）

### 現況
- token 成本顯示為「🪙 預估 AI 用量：約 N ～ M 萬 tokens」(line 631)，來自 `CostEstimateService.TokenRange`（`lowWan()/highWan()`，line 21/26）。對素人＝恐嚇（token 是什麼？會燒多少錢？）。

### 改法
- **成本改成分級**：素人看到「本次預估 AI 用量：**低 / 中 / 高**」。
  - 由現有 `TokenRange.highWan()` 對應分級（純呈現，不改 `CostEstimateService` 計算）：定義 `highWan ≤ T1 → 低`、`≤ T2 → 中`、`否則 高`（T1/T2 為待調的萬-token 門檻，先給保守預設、可在程式常數調整）。
  - 精確「N～M 萬 tokens」與「幾位 AI 工程師」**收進可展開的「詳細」或 tooltip**（開發者仍看得到）。
- **計畫白話化**：確認頁的計畫若呈現為工程任務（建立 frontend / 設計 schema / 實作 API），對素人翻成「正在做哪些頁、會交付什麼」。計畫本體已可編輯（保留）。
- **確認前狀態明示**（強化既有「按確認才開工」）：一句固定文案——「**AI 還沒動工，還沒改任何東西，還沒正式交付；你確認後才開始，且交付的是可預覽草稿，不會自動上線、不會自動收款。**」
- **（選用／後續）線框草圖預覽**：依需求生成低擬真 wireframe，把「還沒動工」轉成「即將展開」。**標為後續**——需要產出能力，且不可做成「看起來像已完成」而違反誠實。本期可先只上文案版。

### 驗收
- 素人在確認頁**看不到裸 token 數字**（除非展開詳細）；成本一眼是低/中/高。
- 確認頁明確讓素人說得出「我知道它接下來要做什麼，也知道這還只是草稿、還沒動工」。

## 3. 把關綁預覽（交付／任務頁，跨頁設計原則）

> 這是 §5 點出的最大盲點的解法，也是本草案的靈魂。

- **素人的審查面 = 線上預覽 + EXPLAINER 白話導覽 + 白話計畫**。交付頁對素人**主打線上預覽**（可點開試用），zip／原始碼／diff／PR／GEP 一律**降級到進階或開發者門**。
- 交付頁誠實三塊（沿用 §1.R2-8）：**現在能看的**（預覽）／**你能自己改的**（店名、品項、價格、文字）／**還不能自動做到的**（上線、收款、LINE、長期保存，除非另設）。
- 「可審查草稿」對素人翻譯成「**這是一個可試看的初版，不是已正式上線的網站**」。
- **治理對素人收斂成一句信任話**（不出現控制台/雜湊鏈/GEP 字樣）：「**AI 不能自己發布，也不能核准自己的成品；你確認前不會正式交付。**」治理控制台 / GEP explorer 維持現狀，只在**開發者門 / 進階**出現。

### 驗收
- 素人完成一次任務的全程，**不需要、也不會被要求**閱讀原始碼、diff、PR 或治理事件。
- 素人若想「再改一點」，入口是面向預覽的白話修改（依成果接續修改已存在），不是面向程式碼。

## 4. 實作順序（小步、IA/文案層級為主）

1. **首頁漸進揭露**：把 mode(import/existing)+zip+projectType+strength 收進「進階／我已有專案」展開區；主區保留描述+Discovery+開始。（純前端 + 既有預設值）
2. **確認頁成本分級**：加 `TokenRange → 低/中/高` 呈現對映；裸數字收進詳細；補「還沒動工/不會上線/不收款」固定文案。
3. **交付頁素人化**：主打線上預覽 + EXPLAINER；zip/PR/治理降級到進階；補誠實三塊 + 一句信任話。
4. **（後續）** 確認頁線框草圖預覽、可解釋「AI 思考脈絡」（「因為你說要賣餅乾，所以加了商品管理」，可點/問/改）——較大、列後續。

### 測試 / 驗收（仿既有 WebUiTest 家族）
- 首頁 HTML：預設不含 `git`/`zip`/`strength` 字樣於主區；進階區存在且含這些。
- 確認頁 HTML：含「低/中/高」其一、含「還沒動工」「不會自動上線」字樣；裸 token 數字在可展開段。
- 既有測試不應因 IA 調整而紅（保留現有 id/欄位語意）。

---

## 5. 邊界與不做（誠實鐵律對齊）

- **不做** 首屏 NLP 意圖自動分流（§5 裁決）。
- **不自編** 金流／精確金額／不存在的客服（Gemini 曾踩線，已濾）。
- 線框草圖預覽**不可**做得像「已完成品」——必須明標「初步構想草圖」。
- 治理／GEP **不對素人隱藏其存在的事實**，只是不放素人首屏——交付頁底部仍可一行「系統已保留製作紀錄，需要時可給工程師檢查」。
