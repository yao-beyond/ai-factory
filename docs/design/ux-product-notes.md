# UX / 產品筆記：系統亮點的三輪深度討論

> 狀態：**討論結論（非實作規格）**。來源：CCG 三輪 UX/產品討論（外部顧問 + Claude 綜合），日期 2026-06-26。
> 顧問註記：本輪原定 Gemini 主跑 UX/產品 lane，但 **Gemini CLI 已被 Google 停用個人 free-tier**（要求遷移 Antigravity），且當前 `omc` 版本未支援 antigravity，故改由 **Codex 戴 UX/產品顧問帽**當外部聲音、Claude 居中收斂（CCG fallback）。視角偏工程顧問味；日後接上純設計 provider 可再跑一次對照。
> 原始 artifacts：`.omc/artifacts/ask/codex-ux-*`（三輪）。

## 0. 一句話結論

**這套系統是「一個工廠、兩種價值」**：

- 對**開發者**：賣點是「可治理、可追責、AI 不能越權」的 AI 變更管制。
- 對**素人**：賣點是「我講不清楚，它幫我收斂成能開工、且誠實知道極限的草稿」。

**核心問題**：這兩種價值目前擠在同一個入口互相傷害——最強的護城河（治理 runtime / GEP / 雜湊鏈）對開發者是賣點，對素人卻是雜訊甚至恐嚇。

---

## 1. 三輪推進軸

### R1 — 全景（亮點排序）
| 排名 | 亮點 | 主要打動 | 護城河判斷 |
|---|---|---|---|
| 1 | AI 變更治理 runtime | 開發者 | 真差異化護城河 |
| 2 | Discovery「幫我想想」+ 能力邊界卡片庫 | 素人 | 潛力護城河 |
| 3 | human-gated 開工前確認 | 素人 | 不錯但可被取代 |
| 4 | 交付「可審查草稿」非假完成品 | 素人 + 開發者 | 半護城河（看執行深度）|
| 5 | 多 AI 並行 + 挑最佳 + Codex 審查 | 開發者 | 需產品化成評選系統才有護城河 |

要點：最強的不是「AI 會寫程式」，是把 AI 寫程式重新包裝成**可控、可審查、可拒絕、可交接**的軟體工廠流程。誠實定位（三層能力邊界）本身就是護城河。

### R2 — 戳破（素人首用逐畫面走查）
以「台中早餐店老闆想做線上預訂」為例，從首頁走到交付的斷點：

1. **首頁**就講製程/治理/PR/main → 素人想「這是工程師用的，我來錯地方」。
2. **三模式（全新/匯入 zip/連 git）**像考題；「連 git」讓人怕按錯弄壞東西 → 系統還沒幫到他，就先讓他覺得不夠格。
3. **Discovery** 該像「店務訪談」（你想少花時間處理哪件事？），不是需求表單（需要資料庫嗎？）。素人卡的不是「沒想法」，是「怕講錯被誤解成巨大工程」。
4. **成品卡片**文案要從「技術能力」改成「可得到的東西 + 不會做什麼」，否則交付時想像落差必失望。
5. **開工前確認頁**：計畫別像工程任務清單；**token 成本對素人是恐嚇**（會不會一直燒錢？），應顯示「低/中/高 或 約 NT$X」，token 細節收進進階。
6. **任務頁即時 log**（npm install / lint failed / retrying）讓素人以為壞了 → 要翻成白話里程碑。
7. **多 AI / 挑最佳** 素人無感 → 一句「系統會試做幾版，留最穩的給你」。
8. **交付**：線上預覽才是素人交付物，**zip 不是**（看到 zip 想「我還要找工程師？」）；要誠實分「現在能看的 / 你能自己改的 / 還不能自動做到的」。

**哪些亮點對素人無感甚至嚇人**：治理 runtime、攔截控制台、雜湊鏈、GEP 證據鏈瀏覽器——對小店老闆都是認知負擔（「我只是想做菜單，為什麼有證據鏈？我要負法律責任嗎？」）。應對素人**隱藏**，收斂成一句信任話。Codex 安全審查可留，但要翻譯成「交付前會自動檢查明顯錯誤；AI 不能自己核准自己的成品」。

**最大單一 UX 風險**：**系統一開始就逼素人進入開發者心智模型**（全新/zip/git/PR/token/治理/審查/證據鏈）。素人第一次成功只需要一件事：從一句白話，走到一個看得見、能試用、知道限制的初版。

### R3 — 收斂（可執行結論）
1. **產品定位（一句話）**：給不懂程式的人把白話想法變成可審查軟體草稿，也讓開發者用 AI 並行產出、但每一步都由人類把關的自架開源 AI 軟體工廠。
2. **入口策略：兩道門**（不是單一智慧分流）。
   - 「**我只有想法**」→ 素人 → Discovery。
   - 「**我已有專案**」→ 開發者 → zip / git / PR / governance。
   - 門名用**任務語言**（非「素人/開發者」），且後續永遠可切換。
   - 風險：別讓它看起來像兩個產品、別讓中間型使用者卡住 → 用任務語言 + 可切換化解。
3. **第一個優先打磨的 UX**：素人首用體驗「**從想法開始 → Discovery 卡片 → 開工前確認**」。可驗收標準：
   - 不需輸入技術名詞，**2 分鐘內**走到開工前確認頁；Discovery 最多 **2 題**、皆生活語言。
   - 卡片清楚寫：做什麼 / 不做什麼 / 會交付什麼。
   - 確認頁白話顯示：需求摘要、預計步驟、成本（低/中/高）、可編輯、確認鈕；按下前明確表達「**AI 還沒動工、還沒改任何東西、還沒正式交付**」。
   - 受測素人能說出：「我知道它接下來要幫我做什麼，也知道這還只是草稿。」
4. **護城河共存（一句話）**：治理對開發者做成可檢查、可追責、可調政策的控制台；對素人只呈現成一句信任——「**AI 已被限制在草稿工廠裡，最後核准永遠是人。**」

## 2. Claude 綜合判斷（同意為主，補充/異議）

- **同意兩道門，但要防它變兩個產品**：後端是同一條 pipeline、同一套治理；改的只是入口框架 + 漸進揭露 + 文案白話化。幾乎全是 IA 與用詞，**低成本，且不違反誠實鐵律**（不是隱藏能力，是不把術語前置）。
- **治理護城河不要對素人關掉，要「翻譯」**：把「攔截控制台 / 雜湊鏈 / fail-closed」對素人收斂成一句安心話。把最硬的工程事實變成素人最想聽的信任話。
- **誠實邊界本身是素人賣點**：卡片上「不會自動收款 / 不會自動傳 LINE」對素人是鬆一口氣，不是功能缺。
- **行銷素材要分眾**：現有 LinkedIn / 對比圖偏開發者口味（治理、fail-closed）；素人側缺一個「兩個問題 → 可預覽草稿」的 30 秒素材——R3 腳本可直接拿來拍，也補上 HANDOFF 提到「還缺一支 60 秒 demo」。
- **要盯的風險**：兩道門別稀釋「human-in-the-loop」這條共同品牌脊椎——兩扇門都該看得到「人類是最後一關」。

## 3. 30 秒素人 Demo 腳本（誠實版，可直接拍）

| 秒 | 畫面 | 旁白 |
|---|---|---|
| 0–3 | 首頁兩道門，點「我只有想法」 | 不是叫你直接跟 AI 賭結果，而是先選你怎麼開工。 |
| 3–8 | Discovery 兩題：「幫誰省時間？→小店老闆」「先解決哪件事？→客人線上預約」 | 不用講資料庫、後端、部署，只回答兩個白話問題。 |
| 8–13 | 3 張成品卡，點「預約表單草稿」 | 卡片就是能力邊界，做不到的不會被包裝成選項。 |
| 13–20 | 開工前確認頁：會做 / 不會做（不收款、不自動上線）/ 成本 / 可編輯 / 確認開工 | AI 先寫計畫、停在這裡。你看懂、改完、確認，它才開工。 |
| 20–25 | 任務流水線：規劃→並行→挑最佳→審查→「產生可審查草稿中」 | AI 可以並行做，但不能碰 main，也不能自己核准交付。 |
| 25–30 | 交付：線上預覽 + 下載 + EXPLAINER，小字「可審查草稿，尚未正式上線」 | 你拿到的是可預覽、可下載、可審查的草稿，不是假裝已上線的產品。 |

鐵律：不可演出假上線 / 假收款；交付畫面必須明標「尚未正式上線」。

## 4. 建議下一步（依投報比排序）

1. **首頁改兩道門 + 文案白話化**（IA/文案，最便宜，打中最大風險）。
2. **素人路徑藏術語**：確認頁 token → 低/中/高 或 約 NT$X；即時 log → 白話里程碑（已有里程碑進度條，差 log 翻譯）；治理收成一句信任話。
3. **拍 R3 的 30 秒素人 demo**（誠實版）。
4. 治理控制台 / GEP explorer 維持現狀，但移到開發者門 / 進階，**別出現在素人首屏**。

> 這些大多是資訊架構 + 文案層級的改動，不需重寫核心系統；治理 runtime 作為開發者護城河保持不變。

## 5. 設計視角對照與裁決（Gemini 純設計 vs Codex 工程顧問）

> 追加於 2026-06-27。前四節由 Codex（工程顧問味）主導；本節引入 **Gemini 純設計視角**（經 `omc-gemini-advisor.mjs` 走 Gemini REST API，模型 gemini-2.5-flash，因 Gemini CLI free-tier 已停用）對撞，再由 Claude 裁決。artifact：`.omc/artifacts/ask/gemini-human-in-the-loop-*`。

### 核心分歧：入口
| | Codex | Gemini |
|---|---|---|
| 主張 | **兩道門**（任務語言、可切換、低認知負擔、誠實、便宜）| **反對兩道門** → 單一溫暖入口 + NLP 語意智慧分流（像夥伴、彈性）|
| 隱憂 | — | 兩道門逼第一屏做判斷、窄化彈性、像工具 |

### 裁決（Claude，偏 Codex、吸收 Gemini 擔憂）
**採「一個溫暖主入口（預設走素人 Discovery）+ 一個安靜的次要『我已有專案/匯入』連結」**——不是兩扇對等門，也**不**採 Gemini 的重型 NLP 自動分流。理由：
- 鐵律是誠實、不靠 AI 亂猜；卡片庫存在的目的就是**刻意不**用自由文字 NLP 過度承諾。首屏 NLP 判斷意圖＝把最克制的東西放回最前面，**第一屏誤判＝最糟第一印象**，與專案氣質衝突。
- 但 Gemini「兩扇對等門增加認知負擔」成立。折中＝回到 Codex R2 原本的「素人預設入口 + 進階收起」（R3 硬化成兩道門、Gemini 拉回，中間值才對）。**此裁決修訂 §1.R3-2 與 §4-1 的「兩道門」說法。**

### Gemini 該照單收的 3 點（與入口之爭無關）
1. **最大設計盲點（Codex 漏）**：素人看不懂程式碼草稿 → 盲目按下一步 → **human-in-the-loop 淪為形式/劇場**，信任反崩。
   - **設計原則（新）**：素人的「把關」一律綁在**線上預覽（看行為）+ EXPLAINER 白話導覽 + 白話計畫**，**永不要素人審原始碼/diff**。（GEP explorer 是開發者版可審查；素人版可審查＝預覽＋白話。）
2. **被低估的機會**：可解釋、可修正的 **「AI 思考脈絡」**（「因為你說要賣餅乾，所以我加了商品管理」，每個推論點可點/問/改）。與「學徒日誌/選評報告」「GEP」同源，值得放大成素人主互動。
3. **確認頁情緒設計**：把「AI 還沒動工」冷訊息改成**低擬真線框草圖預覽**（「這是 AI 初步構想的草圖，準備好生更詳細的？」）→「還沒動工」翻成「即將展開」。便宜有感。

### Gemini 踩到誠實鐵律、已濾掉
- 它自編「約 NT$300、推薦某金流、手續費較低」——系統不串金流、無法保證精確金額，**過度承諾，不採**（成本維持低/中/高）。
- 「客服/社群求助」——若無此資源別寫。

### 一句話收斂
入口用「一個素人主入口 + 安靜進階門」（非兩對等門、非 NLP 猜）；**素人的把關綁在預覽＋白話導覽而非程式碼**——這是本輪最該優先落實的設計原則。
